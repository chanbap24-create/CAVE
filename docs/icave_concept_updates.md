# i cave — 컨셉 업데이트 (2026.04.29)

> Claude Code가 다음 작업 시작 전에 읽어야 할 핵심 변경점.
> 기존 `CLAUDE.md`의 개발 룰은 그대로 유효. 이 문서는 **제품 방향성 변경**을 담음.

---

## 0. 한 줄 요약

i cave는 더 이상 **"와인 SNS 앱"**이 아니다.
**"주류 셀러 + 트레바리식 큐레이션 모임 + 샵 스마트오더"**가 결합된 양면 플랫폼이다.

---

## 1. 핵심 정체성 변경

### Before
- 와인 중심
- SNS + 셀러 + 모임 균형
- 홈 = 일반적인 SNS 피드
- "와인 인스타"에 가까운 느낌

### After
- **주류 전반** (와인 → 위스키/사케/전통주로 확장)
- **셀러가 코어, 트레바리식 큐레이션이 차별점**
- **홈을 없애고 셀러를 첫 진입점으로**
- **SNS는 "행위 중심"만 유지 (피드 죽임)**

---

## 2. 탭 구조 재설계

```
[셀러]      → 내 주류 우주 (홈 역할 흡수)
[Discover]  → 트레바리식 큐레이션 모임 + 시음회 + 가이드
[+]         → 라벨 스캔 등록
[모임]      → 내가 참여한/만든 모임
[프로필]    → 종합 활동
```

**기존 6탭 → 5탭으로 축소.** `(tabs)/index.tsx`(홈 피드)는 제거 대상.

### 셀러 화면이 흡수해야 할 것

지금 `cellar.tsx`는 이미 부분적으로 "내 것 + 남의 것"이 섞여 있음.
이 방향을 더 강화. 위에서 아래로:

1. 셀러 헤로 (내 통계)
2. **다음 모임 알림** (액션 유도) ← 추가 필요
3. 내 픽 (My Picks)
4. 친구들의 셀러 활동 (CellarActivityStrip — 이미 있음)
5. **추천 모임 — 내 셀러 기반 큐레이션** ← 추가 필요
6. 내 와인 목록 (카테고리 탭)
7. 최근 마신 모임 ← 추가 필요

## 3. `posts` 테이블 / 피드 제거

### 결정
인스타식 피드 게시물(post) 죽임. **셀러 등록(collections) 자체가 콘텐츠**가 됨.

### 영향 범위
- `(tabs)/index.tsx` 제거
- `(tabs)/create.tsx` 제거 (라벨 스캔 등록만 남김)
- `posts`, `post_images`, `post_wines` 테이블은 **deprecate** (즉시 drop은 아님, v2에서 정리)
- `usePostSubmit`, `PostCard`, `PopularPosts`, `useMyPosts` 등 관련 훅/컴포넌트 deprecate
- 좋아요/댓글은 **`collections` 단위**로 통일 (이미 `useCollectionSocial` 있음)

### 마이그레이션 전략
1. v1 (지금): 새 cellar 화면에서 posts 의존 제거. 코드는 남겨두되 진입점만 차단.
2. v2: posts 테이블 실제 drop. 마이그레이션 작성.
3. 멘션/알림 시스템은 collections 기반으로 재설계 필요.

## 4. Discover = 트레바리화

### 컨셉
"발견 탭"이 아니라 **"에디토리얼 큐레이션 매거진"**.
인스타 둘러보기처럼 알고리즘 잡동사니가 아니라, 사람이 직접 고른 시즌 클럽.

### 위에서 아래로 큐레이션 강도 약해지는 구조

| 섹션 | 큐레이션 주체 | 결제 강도 |
|---|---|---|
| 시즌 클럽 (히어로) | 샵 + i cave 에디터 | 높음 (선결제) |
| 진행 중인 클럽 | 샵 | 높음 |
| 이번 주 시음회 | 샵 | 중간 |
| 유저 모임 | 유저 | 낮음 |
| 와인/주류 가이드 | i cave 에디터 | - |
| 샵 둘러보기 | - | - |

### Discover에서 SNS적 요소 제외
- 다른 사람 게시물 피드 ❌
- 좋아요/댓글 (Discover 화면 자체에는) ❌
- "팔로잉" 토글 ❌

**Discover는 발견하는 곳이지 소비하는 곳이 아니다.**

## 5. 시즌 클럽 — 새로운 코어 BM

### 구조
- **7주 코스 · 4회 모임**
- 샵 소믈리에가 클럽장 (장기적으로 외부 전문가/검증된 유저 확장)
- 시즌 단위 선결제 (15–30만원/인)
- 매주 다른 와인 시음 + 노트 작성
- **마신 와인이 셀러에 자동 누적**

### 데이터 모델 영향

새 테이블 또는 기존 `gatherings` 확장 필요:

```sql
-- 옵션 A: gatherings에 컬럼 추가
alter table gatherings add column gathering_format
  text check (gathering_format in ('single', 'season_club'));
alter table gatherings add column season_total_sessions smallint;
alter table gatherings add column season_session_index smallint;
alter table gatherings add column season_parent_id bigint references gatherings(id);

-- 옵션 B: 별도 season_clubs 테이블
create table season_clubs (
  id bigint generated always as identity primary key,
  shop_id ... -- 향후 venues 테이블 연동
  host_id uuid references profiles(id),
  title text not null,
  total_sessions smallint default 4,
  duration_weeks smallint default 7,
  price_per_seat numeric(10,0),
  max_members smallint,
  starts_at timestamptz,
  ...
);
-- 그리고 season_club_sessions로 각 회차 관리
```

**아직 결정 안 함.** 시즌 클럽 본격 구현은 v2 시점이라 지금은 스키마만 검토.

## 6. 진정성 (Authenticity) 시스템 — 새로운 코어 자산

### 문제
공병 라벨 스캔만으로 셀러 채울 수 있음 → 셀러가 자랑 자산이 되는 순간 거짓말 가능.

### 해법
**다층 진정성 신호** (이분법 X, 신호의 합 O)

| 신호 | 검증 방법 | 위조 난이도 |
|---|---|---|
| 샵 구매 인증 | 영수증/스마트오더 기록 | ★★★★ |
| 모임 참석 기록 | 여러 명이 같은 자리 증언 | ★★★★★ |
| 테이스팅 노트 | 묘사의 깊이/일관성 | ★★★ |
| 사진 메타데이터 | 직접 촬영·시간·장소 | ★★★ |
| 시간 패턴 | 자연스러운 누적 vs 일괄 등록 | ★★ |

### UI 표현
컬렉션 항목마다 **진정성 배지**가 자연스럽게 붙음:

```
🍷 Château Margaux 2015
   ├─ 🛒 ABC 와인샵에서 구매 (검증됨)
   ├─ 🍽️ "부르고뉴 첫 만남" 모임에서 시음
   ├─ 📝 테이스팅 노트 작성됨
   └─ 📸 직접 촬영
```

라벨만 등록한 와인은 막지 않음. **신호의 양으로 자연스럽게 차이.**

### 호스트 자격 = 진정성 점수 합산
"셀러 30병"이 아니라 다음과 같이:

```
호스트 자격 점수 =
  (검증된 구매 × 3) +
  (모임 참석 × 5) +
  (테이스팅 노트 × 1) +
  ...
```

→ **돈 많이 쓴 사람보다 진짜 활동 많이 한 사람이 호스트가 됨**

### 구현 방향 (단계적)
- v1: `collection_source`(이미 있음) 기반 배지 표시
- v1: 모임 참여 기록과 collection 연결 (`gathering_contributions`로 이미 일부 가능)
- v2: 샵 검증 시스템 (셀프 등록 + 샵 일괄 승인)
- v2: 진정성 점수 계산 함수 (DB function)
- v3: 점수 기반 호스트 자격 게이팅

## 7. 양면 플랫폼 — 샵 측 SaaS 추가

### 핵심 통찰
유저 측 앱만 만들면 가치 절반. **샵 측 운영 도구**까지 만들어야 lock-in.

### 샵 대시보드 (v2 이후)
- 모임 개설/관리
- 단골 명단 (i cave로 알게 된 손님)
- 매출 분석 (모임 → 구매 전환율)
- 재고 노출 (스마트오더 카탈로그)
- 시즌 클럽 운영 도구

### 지금은
모바일 앱만 운영. 샵 대시보드는 v2부터 본격 구현.
다만 **DB 설계 시점부터 양면을 염두**에 둬야 함.

### `venues` 테이블 우선순위 상향
지금 `CLAUDE.md`의 Known Debt에 `venues` 미연동 항목 있음.
**시즌 클럽 + 샵 가맹 모델로 가면 venues는 더 이상 옵션이 아니라 필수.**
v1 후반부에 schema 설계 시작 권장.

## 8. 비즈니스 모델 — 결제/수수료 구조

(개발에 영향 주는 부분만)

### 결제 통합 방향 (v2 목표)
- **스마트오더 통합** (앱 주문 → 매장 픽업)
- **한국 주류통신판매업 신고 필요** — 컴플라이언스 작업 무거움
- 본인인증 + 성인인증 + 미성년자 차단 강제

### 수수료 구조 (앱이 처리해야 할 것)
- 일반 매장 픽업: 3%
- 모임 후 7일 내 구매: 7% (모임-구매 연결 추적 필요)
- 시즌 클럽 결제: 10%
- 신규 유저 첫 구매: 10% (3개월 후 3%로 전환)

### 데이터 모델 영향
- 모임-구매 연결 추적 테이블 필요 (`purchase_attribution` 같은)
- 결제 상태 enum 확장 필요
- CMS 자동이체 (시즌 클럽 정기결제)는 카드 결제와 별도 처리

### 인앱결제(IAP) 회피
- 와인 구매 = 물리적 상품 → IAP 회피 가능
- 모임/시즌 클럽 = 오프라인 서비스 → IAP 회피 가능
- **프리미엄 디지털 구독은 만들지 않음** (만들면 30% 떼임)
- 멤버십 만들 거면 "오프라인 권리"로 재정의 (예: 모임 우선예약권)

## 9. 주류 확장 — 와인이 첫 진입점, 그 이후 단계적

### 코드는 이미 준비됨
```sql
create type spirit_category as enum (
  'wine', 'whiskey', 'cognac', 'sake', 'beer', 'other'
);
```

`drinkCategories.ts` 상수도 4개 카테고리(Wine/Spirit/Traditional/Other)로 단순화 완료.

### 우선순위
- **v1**: 와인 중심 운영 (베타 검증)
- **v2**: 위스키/사케 추가 (어느 쪽이 다음일지 미정)
- **v3**: 전통주

### UI/UX 시사점
- 카테고리 필터는 이미 잘 구현됨
- 와인 전용 용어 줄이기 ("vintage" → 카테고리에 따라 다르게 표시)
- `wine_type` enum은 와인 전용 → 다른 카테고리는 null 허용 (이미 그러함)

## 11. 기존 `CLAUDE.md`와의 관계

기존 룰은 그대로 유효:
- 모듈화, 파일 크기 상한, 매직 넘버 금지

새 기능 추가 시 두 문서를 모두 따른다.

## 12. 보류 — 결정 안 된 것들

이 문서를 만든 시점에 아직 안 정해진 것:

- 회사(까브드뱅) 안에서 사업부로 갈지 별도 법인으로 갈지
- 본인(조성재) 지분/직책/풀타임 합류 시점
- 투자 유치 규모 (3억/5억/7억/10억)
- 와인 다음 카테고리 (위스키 vs 사케 vs 전통주)
- 베타 시점 가맹 샵 모객 전략 디테일

**이 결정들이 코드에 직접 영향을 주진 않음.** 개발은 위 1–10번 우선순위로 진행.
